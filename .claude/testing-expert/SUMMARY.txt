Key Findings from Research:
---------------------------

Project Structure:
- 3 sub-projects: collector (Go), server (Go), client (React - planned)
- Unit tests co-located with source (*_test.go)
- Integration tests in /tests/ directory
- Test utilities in /tests/testutil/

Testing Infrastructure:
- Go standard library testing + testify
- pgx/v5 for PostgreSQL testing
- Temporary test databases with unique names
- Service management utilities
- CLI execution utilities
- Configuration templating

Coverage:
- Go: go test -cover with HTML reports
- Goals: >80% overall, >90% critical, 100% security
- Coverage tracked in CI/CD
- Artifacts retained for 30 days

Quality Checks:
- golangci-lint with 7 enabled linters
- Configuration in .golangci.yml
- Runs on all code including tests
- Part of CI/CD pipeline

Test Patterns Found:
- Table-driven tests extensively used
- Interface-based mocking (no external framework)
- Database tests with automatic cleanup
- Environment variable configuration
- Skip patterns for optional tests

Best Practices Documented:
- Test isolation (independent tests)
- Resource cleanup (defer statements)
- Security testing (validation, authorization)
- Error path testing
- Edge case coverage
- Meaningful test names
- CI/CD integration

CI/CD:
- GitHub Actions workflows for each project
- PostgreSQL 16 service for database tests
- Go 1.23, 1.24, 1.25 matrix testing
- Coverage HTML artifacts uploaded

Environment Variables:
- TEST_AI_WORKBENCH_SERVER: Database connection
- TEST_AI_WORKBENCH_KEEP_DB: Keep test database
- SKIP_DB_TESTS: Skip database tests
- SKIP_INTEGRATION_TESTS: Skip integration tests
